# 夏夜将终 官网 — 代码结构与贡献流程问题报告

> 本报告为**只读分析**，未对项目代码做任何修改。目的：指出当前实现中影响"不太懂代码的乐队成员贡献图片/文字/音频/视频"以及项目可维护性的问题，供你决定是否后续重构。
> 证据来源：根目录 HTML、`assets/js/*.js`、`assets/css/*`、`config.json`、git 提交历史。

## 1. 总体评价

当前架构的核心选择是合理的：**`config.json` 单文件作为内容数据库，`render.js` 在浏览器端读取并注入 DOM**。非技术人员理论上只改 `config.json` 即可。但实际落地中有一系列"为了让界面跑起来"的妥协，导致贡献流程比想象的繁琐且易错。问题分四类：贡献流程、数据模型、前端实现、项目组织。

---

## 2. 贡献流程问题（最影响乐队成员）

### 2.1 "一个作品要改三处"——冗余且易漏 ❗
新增一个翻唱作品需要**同时**往 `discography` + `movies` + `news` 三个数组各加一条（见 `内容贡献指南.md` §3.3）。三处字段还不同（discography 要 cover/original，movies 不要 cover，news 不要 original/cover）。极易漏掉 `movies`，导致首页 MOVIE 列表缺图或缺失。
- 根因：`movies` 与 `discography` 信息高度重叠，本可由 `discography` 派生，却要人工重复录入。
- 建议（未来）：`movies` 改为从 `discography` 自动派生，或合并为单一 `works` 数组加 `type` 字段。

### 2.2 没有结构化"过往演出"入口 ❗
`live` 数组只放了一条"即兴复活 EP02"，且似乎只用于"近期日程"。**已结束的演出没有归档字段**，只能塞进 `biography.history` 的纯文字年表。本次时间线整理出的 11+ 场演出几乎都无处结构化入库（没有 venue / lineup / setlist / poster / 回顾视频 等字段）。
- 建议：新增 `liveHistory`（或 `gigs`）数组，字段 `{date, week, title, venue, host, lineup, setlist:[...], poster, bvid, note}`，并新增一个"过往演出回顾"页面或在 `live.html` 中分"即将/已结束"。

### 2.3 歌词根本无法靠 config.json 添加 ❗
`assets/js/lyrics-render.js` 只有 3 字节（空文件）。`detail-render.js` 的 `renderLyricsDetail` 把歌词正文写死为"歌詞は準備中です"占位，不读取任何 config 字段。也就是说 LYRICS 页面是**装饰性占位**，乐队成员无法自助添加歌词。
- 建议：在 `config.json` 的 discography 条目里加可选 `lyrics` 字段（数组/多段），`renderLyricsDetail` 读取并渲染，无则保留占位。

### 2.4 媒体只能走 B 站，缺本地音频/视频方案
当前音频/视频一律用 B 站 `bvid` 引用。没有 `assets/media/` 目录约定，也没有播放器。若要放本地音频/排练录音/现场音频，目前无标准做法。
- 建议：约定 `assets/media/` 并在 config 加可选 `audio`/`video` 本地路径字段，提供 `<audio>`/`<video>` 渲染分支。

### 2.5 必须起本地服务器，否则页面空白
`render.js` 用 XHR 读 `config.json`，直接双击 HTML 打开会因 `file://` 跨域失败而空白。README 已写 `python -m http.server`，但对非技术成员仍是门槛。
- 建议：可在 README 顶部加"不会用命令行？把整个文件夹拖给会弄的人/或用 VS Code Live Server 插件"的说明；或改用 `<script>` 注入 JSON（把 config 包成 `window.__CONFIG__={...}`）规避跨域，实现双击即可看。

### 2.6 没有校验/预览工具
没有脚本校验 `config.json` 字段完整性（如 discography 缺 cover、news 缺 bvid 但又指向外部）。唯一可用的是 `python -m json.tool`，且因 config.json 带 **UTF-8 BOM** 会报 "Unexpected UTF-8 BOM"（需用 `utf-8-sig`）。这会绊住非技术成员。
- 建议：加一个 `scripts/validate.py` 检查字段+路径是否存在；config.json 去掉 BOM。

## 3. 数据模型问题

### 3.1 成员粒度不一致
`biography.members` 里 Guitar 写成 `"苹果蟹 & 范特西"`——把两位成员塞进一个 name 字段，而其他人是一人一行。后续若要单独展示某成员/头像/社交账号会受限。Keyboard 写成 `"懒仟 / 蛋糕"` 同样合并了两人。
- 建议：拆成两条，或用 `{role, name, sub:[...]}` 支持同 role 多人。

### 3.2 路径写法不统一
`config.json` 内图片路径是相对站点根（如 `assets/img/banner/v01_xxx.jpg`，不带 `/`）。但 `render.js` 里 discography/lyrics 详情链接却用**带前导斜杠**的 `/disco-detail.html?bvid=`、`/lyrics-detail.html?bvid=`（共 2 处前导斜杠）。前导斜杠在 GitHub Pages 项目子路径 `/xyjz/` 下会指向站点根而非项目根，**会 404**（git 历史里也提到过路径适配问题，此处漏网）。
- 建议：统一改成相对路径 `disco-detail.html?bvid=`。

### 3.3 category 取值无约束
`翻唱 / LIVE / 活动` 等值散落各处，LYRICS 页用 `category==='翻唱'` 过滤。一旦有人把"翻唱"写成"翻奏/Cover"就会悄悄失效。建议把 category 收成枚举常量集中定义。

### 3.4 时间格式靠人肉
日期统一 `YYYY.MM.DD` 靠自觉，无校验；`week` 三字母英文也靠自觉。

## 4. 前端实现问题

### 4.1 轮播初始化重复执行
`main.js` 的 `initBannerCarousel` / `initMovieCarousel` 既在 `DOMContentLoaded` 直接调用一次，又在 `contentRendered` 事件里再调用一次（共 3 处引用）。会导致首页 Banner 轮播**重复创建圆点指示器**并可能重复绑定定时器。建议只在 `contentRendered` 后初始化一次。

### 4.2 HTML 用字符串拼接生成，难维护且有 XSS 风险
`render.js` 大量用 `innerHTML = items.map(...).join('')` 拼接，标题等用户内容未转义。若标题里出现 `"` 或 `<` 会破坏结构。也不利于后续维护/加字段。

### 4.3 B 站封面回退 URL 是错的
`getBiliCover` 用 `https://i2.hdslb.com/bfs/archive/{bvid}.jpg`。B 站封面图实际按 `aid`（avid）而非 `bvid` 命名，且域名/路径规则不同，此 URL **基本取不到图**。目前因 `movies` 都能从 `discography` 的 coverMap 命中本地封面而未暴露问题，但一旦 `movies` 有 `bvid` 在 discography 找不到，就会显示坏图。
- 建议：去掉这个回退，改为用本地占位图，或通过 B 站 API 取封面。

### 4.4 头图渲染重复
git 历史显示头图已"改用 `<img>` 标签替代 CSS background-image"，但 `render.js` 的 `renderKeyVisual` 仍在给 `.keyvisual` 设 `backgroundImage`（与 `<img>` 并存），属于遗留无效代码。

### 4.5 SNS 图标用 bilibili 却显示 YouTube 图标
`renderSNS` 用 `<i class="fa fa-youtube-play">` 代表 B 站，语义/视觉均误导。

### 4.6 占位页面"装样子"
- `special.html` 实际渲染的是 discography 前 4 个封面（`renderSpecial` 写死 `slice(0,4)`），并非真正的 SPECIAL 内容。
- `works.html` 恒显示"暂无 Works 信息"。
- `app.html`、`fanletter.html` 存在但内容/用途不明确，README 未说明。
建议要么实装，要么从导航移除，避免误导访客。

## 5. 项目组织问题

### 5.1 静态页命名误导
`feature/livetour2026_ichininsho.html` 实际内容是"即兴复活 EP02"（2025 年），文件名却叫 `livetour2026_ichininsho`（2026 + 一人称？）。会误导维护者。建议按内容命名，如 `feature/2025_ep02_revival.html`。

### 5.2 README 未覆盖 feature/ 与占位页
README 列了页面表但未提 `feature/` 自定义静态页机制，也没标明哪些页是占位。

### 5.3 CSS 体量大且来源混杂
`assets/css/official-service.css`(86KB) 与 `theme.css`(29KB)、`custom.css`(23KB) 叠加，其中 `official-service.css` 应为模板原样搬运，定制改动散在 `custom.css`/`theme.css`，定位样式需跨文件搜索。可考虑加注释分隔"模板原文件 / 请勿改"与"本站定制"。

### 5.4 无构建/测试/CI
无 package.json、无 lint、无自动化校验。对纯静态站可接受，但配合 2.6 的校验脚本会显著降低内容出错率。

## 6. 优先级建议（若要改）

按"对乐队成员贡献体验"收益排序：

1. 高：合并 `discography`+`movies` 派生关系，消除"改三处"；新增结构化过往演出数组与字段。
2. 高：实装 LYRICS（config 加 lyrics 字段 + 渲染）。
3. 中：修前导斜杠链接（3.2）、轮播重复初始化（4.1）、bilibili 封面回退（4.3）。
4. 中：加 `scripts/validate.py` + 去 BOM，README 补"本地预览/占位页/feature"说明。
5. 低：成员粒度、category 枚举、SNS 图标、占位页取舍、静态页改名、CSS 分层注释。

> 以上均为建议，本次未改动任何代码或 config.json。是否执行、按何优先级执行，请你确认后再安排。
